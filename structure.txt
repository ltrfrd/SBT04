SBT04
School bus transport and operations system with a workflow-first configuration
surface, a stable dispatch engine, and a stable attendance/reporting layer.

===============================================================================
WORKFLOW FLOW
===============================================================================

Operator workflow target:

Route -> Run -> Stop -> Students

Supporting configuration steps:
- assign schools to route
- assign driver to route
- assign bus to route

Important rule:
- operators should work in routes, runs, stops, and students
- operators should not think in StudentRunAssignment rows
- runtime assignment truth is stop-based and explicit
- no assignment exists without stop context

===============================================================================
INTERNAL VS OPERATOR
===============================================================================

Operator-facing layer:
- Route
- Run
- Stop
- Student
- School identity via school_id + school_name
- Route-school assignment
- Route-driver assignment

Internal runtime layer:
- StudentRunAssignment
- pickup/dropoff state
- onboard state
- running board source data
- attendance/report derivation state

StudentRunAssignment warning:
- INTERNAL ONLY

Workflow context endpoints:
- POST /routes/{route_id}/runs
- POST /runs/{run_id}/stops
- POST /runs/{run_id}/stops/{stop_id}/students
- POST /runs/{run_id}/stops/{stop_id}/students/bulk
- stop-context student creation creates StudentRunAssignment internally
- /runs/start is operational runtime only
- /runs/start starts existing prepared runs only
- prepared run means stops already exist
- prepared run means at least one runtime student assignment already exists
- /runs/start does not create students
- /runs/start does not create stops
- /runs/start does not auto-create StudentRunAssignment rows
- StudentRunAssignment is the runtime + planning bridge for the selected run/stop
- assignment create is stop-context only: POST /runs/{run_id}/stops/{stop_id}/students
- assignment contextual remove is stop-context only: DELETE /runs/{run_id}/stops/{stop_id}/students/{student_id}
- removes the student from the selected run-stop planning context without deleting the student record entirely
- planned run only
- assignment update is maintenance only: PUT /students/{student_id}/assignment
- planned run only
- DELETE /students/{student_id} is full system-wide student deletion, not normal run-stop workflow removal
- assignment delete is blocked directly: DELETE /student-run-assignments/{id}
- blocked direct delete points to canonical contextual delete: DELETE /runs/{run_id}/stops/{stop_id}/students/{student_id}
- NOT FOR OPERATOR WORKFLOW
- internal runtime engine
- still exists in the backend
- direct POST /student-run-assignments/ is blocked
- read endpoints remain available for compatibility/debug visibility
- should not be used as the main UI planning object

backend/ architecture note:
- Route is the planning container.
- Run belongs to route.
- Stop belongs to run.
- Students are managed from explicit stop context in the preferred workflow.
- StudentRunAssignment remains the internal runtime engine.
- app.py is now bootstrap-focused.
- database tables are initialized in FastAPI lifespan, not at import time.
- server-rendered page routes live in backend/routers/web_pages.py.
- session/auth endpoints live in backend/routers/auth.py.
- the GPS websocket endpoint and active connection state live in backend/routers/ws.py.
- route-first driver workspace helpers live in backend/utils/driver_workspace.py.
- Bus is now a standalone entity.
- Route may point to one current bus through route.bus_id.
- route no longer owns vehicle identity or route-level vehicle fallback in user-facing layers.
- bus is the only user-facing source of unit_number.
- bus is the only displayed source of capacity, size, and plate data.
- route creation/update does not require vehicle identity data in the preferred workflow.
- if no bus is assigned, no vehicle details are shown.
- route-driver assignment supports one primary/default driver and one active/current driver.
- primary and active may differ during temporary replacement coverage.
- active route-driver assignment remains the route-level source used for run start.
- route-driver start_date and end_date are legacy/admin/history fields only and are not authoritative for live routing.
- /runs/start starts prepared runs only; it does not create students, stops, or runtime assignment rows.
- stop-context student creation is the preferred path that creates runtime assignment rows.
- StudentRunAssignment is the internal runtime + planning bridge, not the normal operator planning object.
- school mobile attendance pages currently map as:
- /reports/school/{school_id}/mobile -> school_attendance_routes.html
- /reports/school/{school_id}/mobile/route/{route_id} -> school_attendance_runs.html
- /reports/school/{school_id}/mobile/run/{run_id} -> school_mobile_report.html
- attendance templates use TemplateResponse(request, template_name, context).

Real workflow-related endpoint note:
- run router already contains stop-context student endpoints
- single add: POST /runs/{run_id}/stops/{stop_id}/students
- single remove: DELETE /runs/{run_id}/stops/{stop_id}/students/{student_id}
- bulk add:   POST /runs/{run_id}/stops/{stop_id}/students/bulk
- student_run_assignment router still exists for compatibility/internal read visibility
- direct assignment update intentionally lives in student router maintenance endpoint: PUT /students/{student_id}/assignment
- direct POST /student-run-assignments/ is blocked and points callers to stop-context student creation
- direct DELETE /student-run-assignments/{id} is blocked and points callers to the canonical contextual delete workflow: DELETE /runs/{run_id}/stops/{stop_id}/students/{student_id}
- legacy direct creates still exist:
- POST /runs/ for planned-run compatibility
- POST /stops/ for stop compatibility; still planned-run-only when it targets a run
- POST /students/ for student compatibility; still planned-run-only when it targets a stop's run

===============================================================================
ARCHITECTURE NOTES
===============================================================================

SBT04 is not a rewrite.

Protected layers:
- dispatch layer is stable and protected
- attendance layer is stable and protected

Active layer:
- configuration/workflow layer is the current focus

Current operator direction:
- create route
- route creation does not require bus/unit data
- assign schools to route
- assign driver to route
- assign bus to route
- create runs inside route
- create stops inside run
- add students from stop context
- start prepared runs through /runs/start only after stops and student assignments exist

Key reality from the current repo:
- templates remain the main UI
- React frontend exists as a secondary scaffold
- StudentRunAssignment remains required internally
- workflow improvements should be additive and backward compatible
