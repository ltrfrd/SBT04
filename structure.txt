SBT04
School bus operations backend with route/run management, flexible runtime stop execution,
bus-level Pre-Trip Inspection, per-run Post-Trip Inspection, run-start and selected run-end
safety enforcement, route primary/active bus control, and persistent dispatch alerts.

===============================================================================
IMPLEMENTED SYSTEM NOTES
===============================================================================

- Driver workflow keeps Start Run locked until today's selected-route bus Pre-Trip is valid
- Driver workflow can correct the same-day invalid Pre-Trip record inline before Start Run unlocks again
- Driver workflow keeps End Run locked until active-run Post-Trip Phase 2 is complete
- One Pre-Trip Inspection exists per bus per day
- Pre-Trip Inspection is stored by bus_id and remains API-compatible with bus_number and license_plate
- Pre-Trip Inspection create and correction flows sync dispatch alerts for major defects and fit_for_duty = no
- Missing Pre-Trip Inspection alerts are currently triggered through the run-start flow and supported by alert helpers
- Active bus is authoritative for run-start safety checks
- assign_bus sets the active bus and seeds the primary bus when the route has no primary bus yet
- set_primary_bus updates the route's default/base bus
- set_active_bus switches the current operational bus
- restore_primary_bus moves the active bus back to the primary bus and can store a clearance note
- Runs store scheduled_start_time and scheduled_end_time in addition to actual start_time and end_time
- Run start is blocked when no valid Pre-Trip Inspection exists for today's active bus, when fit_for_duty is no, or when any major defect exists
- Post-Trip Inspection exists per run with Phase 1 and Phase 2 flow
- `POST /runs/end` and `POST /runs/end_by_driver` require Post-Trip Inspection Phase 2
- `POST /runs/{run_id}/complete` remains legacy-compatible and does not currently enforce Post-Trip Inspection Phase 2
- The driver workspace page now renders the active-run Post-Trip workflow and only unlocks End Run after Phase 2 completion
- Post-Trip Inspection Phase 2 syncs the urgent major-defect alert when exterior_status is major
- Post-Trip Inspection persists `last_known_lat`, `last_known_lng`, `last_location_update_at`, and `last_driver_activity_at` from the existing websocket GPS stream when a post-trip record already exists
- Post-Trip Inspection decision-layer fields support pending/completed/inactive classification
- Post-Trip Inspection neglect alerts are currently read-triggered from `GET /runs/{run_id}/posttrip`, which also stamps `neglect_flagged_at` when the record first reaches suspected neglect state
- Runtime stop execution remains flexible through arrive_stop, next_stop, pickup, and dropoff endpoints
